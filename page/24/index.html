应用监控

前面我们和大家介绍了 Prometheus 的数据指标是通过一个公开的 HTTP(S) 数据接口获取到的，我们不需要单独安装监控的 agent，只需要暴露一个 metrics 接口，Prometheus 就会定期去拉取数据；对于一些普通的 HTTP 服务，我们完全可以直接重用这个服务，添加一个 /metrics 接口暴露给 Prometheus；而且获取到的指标数据格式是非常易懂的，不需要太高的学习成本。

现在很多服务从一开始就内置了一个 /metrics 接口，比如 Kubernetes 的各个组件都直接提供了数据指标接口，有一些服务即使没有原生集成该接口，也完全可以使用一些 exporter 来获取到指标数据，比如 mysqld_exporter、node_exporter，这些 exporter 就有点类似于传统监控服务中的 agent，作为服务一直存在，用来收集目标服务的指标数据然后直接暴露给 Prometheus。

指标类型

从存储上来讲所有的监控指标都是相同的，但是在不同的场景下这些指标又有一些细微的差异。 例如，在 Node Exporter 返回的样本中指标 node_load1 反应的是当前系统的负载状态，随着时间的变化这个指标返回的样本数据是在不断变化的。而指标 node_cpu_seconds_total 所获取到的样本数据却不同，它是一个持续增大的值，因为其反应的是 CPU 的累计使用时间，从理论上讲只要系统不关机，这个值是会一直变大。

为了能够帮助用户理解和区分这些不同监控指标之间的差异，Prometheus 定义了 4 种不同的指标类型：Counter（计数器）、Gauge（仪表盘）、Histogram（直方图）、Summary（摘要）。

在 node-exporter（后面会详细讲解）返回的样本数据中，其注释中也包含了该样本的类型。例如：

Counter

Counter (只增不减的计数器) 类型的指标其工作方式和计数器一样，只增不减，所以它对于存储诸如服务的 HTTP 请求数量或使用的 CPU 时间之类的信息非常有用。常见的监控指标，如 http_requests_total、node_cpu_seconds_total 都是 Counter 类型的监控指标。

可能你会觉得一直增加的数据没什么用处，了解服务从开始有多少请求有什么价值吗？但是需要记住，每个指标都存储了时间戳的，所有你的 HTTP 请求数现在可能是 1000 万，但是 Prometheus 也会记录之前某个时间点的值，我们可以去查询过去一个小时内的请求数，当然更多的时候我们想要看到的是请求数增加或减少的速度有多快，因此通常情况对于 Counter 指标我们都是去查看变化率而不是本身的数字。PromQL 内置的聚合操作和函数可以让用户对这些数据进行进一步的分析，例如，通过 rate() 函数获取 HTTP 请求的增长率：

Gauge

与 Counter 不同，Gauge（可增可减的仪表盘）类型的指标侧重于反应系统的当前状态，因此这类指标的样本数据可增可减。常见指标如 node_memory_MemFree_bytes（当前主机空闲的内存大小）、node_memory_MemAvailable_bytes（可用内存大小）都是 Gauge 类型的监控指标。由于 Gauge 指标仍然带有时间戳存储，所有我们可以看到随时间变化的值，通常可以直接把它们绘制出来，这样就可以看到值本身而不是变化率了，通过 Gauge 指标，用户可以直接查看系统的当前状态。

这些简单的指标类型都只是为每个样本获取一个数字，但 Prometheus 的强大之处在于如何让你跟踪它们，比如我们绘制了两张图，一个是 HTTP 请求的变化率，另一个是分配的 gauge 类型的实际内存，直接从图上就可以看出这两个之间有一种关联性，当请求出现峰值的时候，内存的使用也会出现峰值，但是我们仔细观察也会发现在峰值过后，内存使用量并没有恢复到峰值前的水平，整体上它在逐渐增加，这表明很可能应用程序中存在内存泄露的问题，通过这些简单的指标就可以帮助我们找到这些可能存在的问题。

Histogram 和 Summary

除了 Counter 和 Gauge 类型的监控指标以外，Prometheus 还定义了 Histogram 和 Summary 的指标类型。Histogram 和 Summary 主用用于统计和分析样本的分布情况。

在大多数情况下人们都倾向于使用某些量化指标的平均值，例如 CPU 的平均使用率、页面的平均响应时间，这种方式也有很明显的问题，以系统 API 调用的平均响应时间为例：如果大多数 API 请求都维持在 100ms 的响应时间范围内，而个别请求的响应时间需要 5s，那么就会导致某些 WEB 页面的响应时间落到中位数上，而这种现象被称为长尾问题。

为了区分是平均的慢还是长尾的慢，最简单的方式就是按照请求延迟的范围进行分组。例如，统计延迟在 0~10ms 之间的请求数有多少而 10~20ms 之间的请求数又有多少。通过这种方式可以快速分析系统慢的原因。Histogram 和 Summary 都是为了能够解决这样的问题存在的，通过 Histogram 和Summary 类型的监控指标，我们可以快速了解监控样本的分布情况。

Summary

摘要用于记录某些东西的平均大小，可能是计算所需的时间或处理的文件大小，摘要显示两个相关的信息：count（事件发生的次数）和 sum（所有事件的总大小），如下图计算摘要指标可以返回次数为 3 和总和 15，也就意味着 3 次计算总共需要 15s 来处理，平均每次计算需要花费 5s。下一个样本的次数为 10，总和为 113，那么平均值为 11.3，因为两组指标都记录有时间戳，所以我们可以使用摘要来构建一个图表，显示平均值的变化率，比如图上的语句表示的是 5 分钟时间段内的平均速率。

Histogram

摘要非常有用，但是平均值会隐藏一些细节，上图中 10 与 113 的总和包含非常广的范围，如果我们想查看时间花在什么地方了，那么我们就需要直方图了。直方图以 bucket 桶的形式记录数据，所以我们可能有一个桶用于需要 1s 或更少的计算，另一个桶用于 5 秒或更少、10 秒或更少、20 秒或更少、60 秒或更少。该指标返回每个存储桶的计数，其中 3 个在 5 秒或更短的时间内完成，6 个在 10 秒或更短的时间内完成。Prometheus 中的直方图是累积的，因此所有 10 次计算都属于 60 秒或更少的时间段，而在这 10 次中，有 9 次的处理时间为 20 秒或更少，这显示了数据的分布。所以可以看到我们的大部分计算都在 10 秒以下，只有一个超过 20 秒，这对于计算百分位数很有用。

指针的本质：内存地址的引用

在 Go 语言中，指针 (Pointer) 存储的是一个变量在内存中的地址。而不是变量的实际值本身。你可以把它想象成一个“门牌号”，通过这个门牌号，你可以找到存储在特定房间里的“人”（数据）。

Go 语言中指针的意义 (Why Pointers in Go?)

虽然 Go 语言在设计上强调简洁和安全，并尝试隐藏底层的内存细节，但指针仍然是其强大和灵活的重要组成部分。其意义主要体现在以下几个方面：

  1. 允许函数修改调用者作用域中的变量：

    • Go 语言的函数参数传递默认是值传递。这意味着当您将一个变量传递给函数时，函数会收到该变量的一个副本。函数内部对副本的任何修改都不会影响到原始变量。
    • 然而，如果将变量的指针传递给函数，函数接收到的是变量地址的副本。通过这个地址，函数可以直接访问并修改原始变量在内存中的值。这是 Go 语言中实现“传引用”效果（即修改原数据）的唯一方式。
    • 意义： 使得构建像 fmt.Scanf 这样能够将输入直接写入到变量的功能成为可能；允许实现一些需要改变外部状态的函数。

    示例：

  2. 避免大数据结构的拷贝开销，提高性能：

    • 当您传递大型数据结构（如大型结构体 struct）作为函数参数时，如果采用值传递，Go 运行时会为整个结构体创建一个完整的副本。这会消耗大量的内存和 CPU 时间，尤其是在频繁调用时。
    • 通过传递结构体的指针，您只需要拷贝一个内存地址（通常为 4 或 8 字节），而不是整个结构体的数据。这大大减少了内存拷贝的开销，从而提升了程序的性能。
    • 意义： 这是优化 Go 应用程序性能的关键策略之一，特别是在处理网络请求中的数据、数据库行、或大型业务对象时。

    示例：

  3. 表示可选参数或某个值不存在的情况：

    • 在 Go 语言中，基本数据类型（如 int、string、bool）都有其零值（0、""、false）。很多时候，我们无法区分一个零值是真实的业务值，还是因为变量未被赋值而得到的默认值。
    • 通过使用指针类型，我们可以表示一个值是存在（指针不为 nil）或不存在（指针为 nil）。
    • 意义： 常见于 API 设计中，比如一个 JSON 请求体中的某些字段是可选的，或者数据库中某个字段允许为空。*int、*string 就可以用来表示这些可选字段。

  4. 实现某些特定的数据结构：

    • 虽然 Go 语言的标准库很少直接暴露指针给开发者（如链表、树结构通常通过值类型和结构体内部的引用来封装），但在构建某些底层或自定义数据结构时，指针是必不可少的。例如，链表的 Next 字段就是一个指向下一个节点的指针。
    • 意义： 允许开发者在需要时，灵活地构建复杂的数据结构。

Go 语言中指针的特性 (与 C/C++ 的异同)

虽然 Go 有指针，但与 C/C++ 等语言中的指针有显著不同，这些差异也体现了 Go 的设计哲学：安全和简化。

  • 没有指针算术： Go 语言不允许对指针进行加减运算（即 ptr + 1 这样的操作），这极大地减少了内存访问越界和段错误的风险。
  • 垃圾回收： Go 语言有运行时垃圾回收机制，开发者无需手动分配和释放内存（malloc/free）。这大大降低了内存泄漏和野指针（dangling pointer）的风险。
  • 结构体字段的自动解引用： 如果 p 是一个结构体指针，访问其字段可以使用 p.Field，而不是 (*p).Field。Go 编译器会自动为您进行解引用。
  • 默认值： 指针的零值（默认值）是 nil，表示它不指向任何有效的内存地址。

何时使用指针？

  • 当你需要修改函数外部的变量时。
  • 当你需要传递大型数据结构，以避免不必要的拷贝开销时。
  • 当你需要表示一个值是可选的，或者可能不存在时（例如，处理 JSON 或数据库中的可空字段）。
  • 当你为类型定义方法时，如果方法需要修改接收者，那么接收者就应该是一个指针类型 (func (s *MyStruct) MyMethod())。

何时避免使用指针？

  • 对于基本类型（int、string、bool 等）和小结构体，通常直接按值传递更安全、清晰，且性能开销可以忽略不计。
  • 对于 Go 内置的引用类型：slice（切片）、map（映射）、chan（通道）。这些类型本身就包含了指向底层数据的指针，因此它们在作为函数参数传递时，也表现出类似“传引用”的行为（函数内部修改会影响外部）。你通常不需要再传递它们的指针（例如 *[]int 是不必要的，直接 []int 即可）。

核心思想： Go 鼓励使用值语义，只有在明确需要上述指针的“意义”时才使用它们。滥用指针会使代码变得复杂，并可能隐藏问题。理解 Go 的值传递哲学和指针的精确用途，是编写 idiomatic 和高性能 Go 代码的关键。

指针接收与值接收

发表于 2025-09-11 分类于 go
本文字数： 3.7k 阅读时长 ≈ 3 分钟

您好，这是一个关于 Go 语言中方法集（Method Set）和接收器（Receiver）的重要问题，对于理解 Go 语言的面向对象特性至关重要。

核心答案

让我们直接给出结论，然后进行详细解释：

  1. 非接口的任意类型 T 能调用 *T 的方法吗？

    • 取决于 T 的“可寻址性 (addressability)”。如果 T 是一个可寻址的值（例如，一个变量），那么是的，它可以调用 *T 的方法，Go 会自动取其地址。如果 T 是一个不可寻址的值（例如，一个字面量或临时值），那么不能直接调用 *T 的方法。

  2. 反过来，*T 能调用 T 的方法吗？

    • 是的，非接口的任意类型 *T 总是可以调用 T 的方法。 Go 会自动解引用。

详细解释：Go 语言的方法集规则

Go 语言的方法集定义了给定类型可以调用的所有方法。理解这个概念是关键。

方法接收器的两种形式：

  1. 值接收器 (Value Receiver): func (t T) MethodName() { ... }

    • 使用值接收器的方法，其作用在类型 T 的一个副本上。
    • 如果方法内部修改了 t，这些修改不会影响到原始值。

  2. 指针接收器 (Pointer Receiver): func (t *T) MethodName() { ... }

    • 使用指针接收器的方法，其作用在类型 T 的一个指针指向的实例上。
    • 如果方法内部修改了 *t，这些修改会影响到原始值。

场景一：非接口的任意类型 T 调用 *T 的方法

规则：

  • 类型 T 的方法集包含所有使用 T 作为值接收器的方法。
  • 如果 T 是可寻址的，那么它还可以调用所有使用 *T 作为指针接收器的方法。 Go 会自动将 &T 传递给方法。
  • 如果 T 是不可寻址的，那么它不能调用使用 *T 作为指针接收器的方法。

什么使一个值“可寻址”？

  • 变量（包括通过 var 或 := 声明的）
  • 结构体字段
  • 数组元素
  • 切片元素
  • 指向某物的指针解引用（例如 *ptr）

什么使一个值“不可寻址”？

  • 字面量（例如 MyStruct{}，10，"hello"）
  • 常量
  • 函数调用的结果（除非返回的是指针）
  • 类型转换的结果（例如 int(float64var)）

场景二：非接口的任意类型 *T 调用 T 的方法

规则：

  • 类型 *T 的方法集包含所有使用 T 作为值接收器的方法。 Go 会自动解引用 *T。
  • 类型 *T 的方法集也包含所有使用 *T 作为指针接收器的方法。

总结方法集规则

  • 对于类型 T： 其方法集包含所有以 T 为接收器的方法。
  • 对于类型 *T： 其方法集包含所有以 T 为接收器的方法和所有以 *T 为接收器的方法。

这也解释了为什么：

  • 一个 *T 的变量可以满足一个要求 T 类型的方法的接口。
  • 一个 T 的变量只有在可寻址的情况下才能满足一个要求 *T 类型的方法的接口。如果 T 是不可寻址的（如字面量），它就不能满足这样的接口。

为什么 Go 要这样设计？

  1. 便利性： 减少了在代码中手动添加 & 和 * 操作符的需要，使代码更简洁、更易读。
  2. 安全性： 对于值接收器，Go 通过复制确保了原始数据的不可变性，除非你显式地传递指针。
  3. 效率： 对于大型结构体，指针接收器可以避免昂贵的数据拷贝。

实践建议

  • 一致性： 在一个类型上定义方法时，尽量保持接收器类型的一致性。如果一个类型的大多数方法都是指针接收器，那么就都用指针接收器，反之亦然。这可以帮助避免混淆。

  • 选择依据：

    • 需要修改接收器的数据时，或接收器是大型结构体时： 使用指针接收器。
    • 接收器是小型结构体、基本类型，且不希望被方法修改其数据时： 使用值接收器（它会创建一个副本）。

数据模型

在开始学习 PromQL 的知识之前，我们先重新来熟悉下 Prometheus 的数据模型

时间序列

Prometheus 会将所有采集到的样本数据以时间序列的方式保存在内存数据库中，并且定时保存到硬盘上。时间序列是按照时间戳和值的序列顺序存放的，我们称之为向量(vector)，每条时间序列通过指标名称(metrics name)和一组标签集(labelset)命名。如下所示，可以将时间序列理解为一个以时间为 X 轴的数字矩阵：

在时间序列中的每一个点称为一个样本（sample），样本由以下三部分组成：

  • 指标(metric)：指标名和描述当前样本特征的标签集合
  • 时间戳(timestamp)：一个精确到毫秒的时间戳
  • 样本值(value)： 一个 float64 的浮点型数据表示当前样本的值

在形式上，所有的指标都通过如下格式表示：

  • 指标的名称可以反映被监控样本的含义（比如，http*request_total 表示当前系统接收到的 HTTP 请求总量）。指标名称只能由 ASCII 字符、数字、下划线以及冒号组成并必须符合正则表达式[a-zA-Z*:]a-zA-Z0-9*:]*。
  • 标签(label)反映了当前样本的特征维度，通过这些维度 Prometheus 可以对样本数据进行过滤，聚合等。标签的名称只能由 ASCII 字符、数字以及下划线组成并满足正则表达式 [a-zA-Z_][a-zA-Z0-9_]*。

指标名称

在 TSDB 内部，指标名称也只是一个特殊的标签，标签名为 __name__，由于这个标签在 PromQL 中随时都会使用，所以在使用 PromQL 查询的时候就被单独写在了标签列表前面了。另外像 method="" 这样的空标签在 Prometheus 种相当于一个不存在的标签，在 Prometheus 代码里面是明确地剥离了空标签的，并不会存储它们。

类型

每个不同的 metric_name和 label 组合都称为时间序列，在 Prometheus 的表达式语言中，表达式或子表达式包括以下四种类型之一：

  • 瞬时向量（Instant vector）：一组时间序列，每个时间序列包含单个样本，它们共享相同的时间戳。也就是说，表达式的返回值中只会包含该时间序列中的最新的一个样本值。而相应的这样的表达式称之为瞬时向量表达式。
  • 区间向量（Range vector）：一组时间序列，每个时间序列包含一段时间范围内的样本数据，这些是通过将时间选择器附加到方括号中的瞬时向量（例如[5m]5 分钟）而生成的。
  • 标量（Scalar）：一个简单的数字浮点值。
  • 字符串（String）：一个简单的字符串值。

所有这些指标都是 Prometheus 定期从 metrics 接口那里采集过来的。采集的间隔时间的设置由 prometheus.yml 配置中的 scrape_interval 指定。最多抓取间隔为 30 秒，这意味着至少每 30 秒就会有一个带有新时间戳记录的新数据点，这个值可能会更改，也可能不会更改，但是每隔 scrape_interval 都会产生一个新的数据点。

数组和切片

发表于 2025-09-11 分类于 go
本文字数： 2.8k 阅读时长 ≈ 3 分钟

好的，数组（Array）和切片（Slice）是 Go 语言中非常核心的数据结构，它们的区别是 Go 开发者必须深刻理解的知识点。作为后端专家，我将从定义、内存、行为和使用场景四个方面，为您详细剖析它们的异同。

核心观点速览

  • 数组 (Array)：定长的、值类型的数据结构。它是一个承载固定数量元素的容器，长度是其类型的一部分。可以把它想象成一个“固定长度的工具箱”，格子数量焊死了，不能多也不能少。
  • 切片 (Slice)：动态的、引用类型的数据结构。它是一个对底层数组的“视图”或“窗口”。可以把它想象成一个“可调节大小的取景框”，你可以移动它，也可以调整它的大小（容量范围内），它框住的内容来自于一张大照片（底层数组）。

在 Go 的实际开发中，99% 的情况下你都应该优先使用切片。

对比表格：异同概览

特性 数组 (Array) 切片 (Slice)
定义 var a [5]int var s []int
长度 (Length) 固定的，在编译时确定。 可变的，可以在运行时通过 append 等操作改变。
类型特征 长度是类型的一部分。[5]int 和 [10]int 是完全不同的两种类型。 长度不是类型的一部分。所有 int 类型的切片都属于 []int 这个类型。
传递行为 值类型 (Value Type)。作为函数参数时，会完整拷贝整个数组。 **引用类型 (Reference Type)**。作为函数参数时，传递的是一个描述符的拷贝，指向同一个底层数组。
内存原理 一段连续的内存，存储着数据本身。 一个三元组**描述符 (Header)，包含：指针、长度 (len)、容量 (cap)**。
初始化 必须指定长度。 可以用 make、字面量或对数组/切片进行“切片”操作来创建。

深入解析：不同点

1. 长度 (Fixed vs. Dynamic) - 最核心的区别

  • 数组 (Array): 一旦声明，其长度就无法改变。

  • 切片 (Slice): 长度是动态的，可以随时通过 append 来增加。

2. 类型标识 (Type Identity)

  • 数组 (Array): 数组的长度是其类型签名的一部分。

  • 切片 (Slice): 切片的类型只由其元素类型决定。

3. 传递行为 (Value vs. Reference)

这是导致行为差异的关键。

  • 数组 (Array) - 值传递，发生拷贝：
    当数组作为函数参数时，函数接收到的是该数组的一个完整副本。在函数内部对副本的修改，不会影响到原始数组。

  • 切片 (Slice) - 引用传递（效果上），共享底层数据：
    当切片作为函数参数时，传递的是切片头（header）的副本，但这个副本中的指针和原始切片的指针指向同一个底层数组。因此，在函数内部通过该切片修改元素，会影响到原始切片。

深入解析：相同点与关联

1. 底层内存布局 (Underlying Memory)

切片是建立在数组之上的。 这是它们之间最根本的联系。

一个切片不拥有任何数据。它只是对一个底层数组的描述。这个描述就是一个切片头（Slice Header），结构如下：

  • 指针 (Pointer): 指向底层数组中该切片第一个元素的位置。
  • 长度 (Len): 切片中包含的元素个数。len(s)。
  • 容量 (Cap): 从切片的起始位置到底层数组末尾的元素个数。cap(s)。

上图清晰地展示了，一个数组可以被多个切片共享。

2. 元素访问与迭代

无论是数组还是切片，访问元素和进行遍历的语法是完全相同的。

实践指南：何时使用哪个？

绝大多数情况，请使用切片 (Slice)。
切片是 Go 语言的惯用法（idiomatic Go）。它的动态性和灵活性使其成为处理集合数据的首选。

  • 函数参数和返回值
  • 需要动态添加或删除元素
  • 构建任何非固定大小的列表

在极少数特定场景下，才使用数组 (Array)。

  1. 当你需要精确控制内存布局时：
    例如，在与其他语言（如 C）进行交互（FFI），或者在实现某些底层网络协议时，你可能需要一个精确大小、无额外开销（没有 header）的内存块。

  2. 当数据量小且固定，且希望它在栈上分配时：
    一个小的数组更有可能被分配在栈上，避免了堆分配和 GC 的开销。这在一些性能极端敏感的场景下可能有意义。

总结

  • 相同点: 都是同构（存储相同类型元素）的连续内存序列，访问和迭代语法一致。
  • 不同点: 数组定长、值类型；切片动态、引用类型。
  • 核心关系: 切片是数组的视图，它通过一个轻量级的描述符（指针、长度、容量）来管理对底层数组的访问。
  • 最佳实践: 除非有非常明确的理由，否则总是优先使用切片。它是 Go 语言更强大、更灵活、更常用的工具。

备份

KubeBlocks 还提供备份恢复功能，以确保数据的安全性和可靠性。KubeBlocks 采用物理备份的方式，将数据库中的物理文件作为备份对象。你可以根据实际需求，选择对应的方式按需或定时备份集群数据。

  • 按需备份：根据不同的备份选项，按需备份可以进一步分为备份工具备份和快照备份两种。
    • 备份工具备份：可使用数据库产品的备份工具，如 MySQL XtraBackup 和 PostgreSQL pg_basebackup。KubeBlocks 支持为不同的数据产品配置备份工具。
    • 快照备份：如果你的数据存储在支持快照的云盘中，你可以通过快照创建数据备份。快照备份通常比备份工具备份更快，因此推荐使用。
  • 定时备份：可指定保留时间、备份方法、时间等参数来自定义备份设置。

KubeBlocks 的备份恢复功能依赖于 BackupRepo，在使用完整的备份恢复功能之前，首先需要配置 BackupRepo。BackupRepo 是备份数据的存储仓库，支持配置 OSS（阿里云对象存储）、S3（亚马逊对象存储）、COS（腾讯云对象存储）、GCS（谷歌云对象存储）、OBS（华为云对象存储）、MinIO 等兼容 S3 协议的对象存储作为备份仓库，同时支持 K8s 原生的 PVC 作为备份仓库。

用户可以创建多个 BackupRepo 以适应不同的场景。例如，根据不同的业务需求，可以把业务 A 的数据存储在 A 仓库，把业务 B 的数据存储在 B 仓库，或者可以按地区配置多个仓库以实现异地容灾。在创建备份时，你需要指定备份仓库。你也可以创建一个默认的备份仓库，如果在创建备份时未指定具体的仓库，KubeBlocks 将使用此默认仓库来存储备份数据。

如果你没有使用云厂商的对象存储，可在 Kubernetes 中部署开源服务 MinIO，用它来配置 BackupRepo。如果你正在使用云厂商提供的对象存储服务，可以直接使用。我们这里可以使用 MinIO 来配置一个 BackupRepo，先在 kb-system 命名空间中安装 MinIO。
