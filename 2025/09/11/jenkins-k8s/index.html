Jenkins 与 Kubernetes 集成的核心价值

  1. 弹性伸缩的构建环境： Jenkins 可以根据构建需求动态创建 Kubernetes Pod 作为 Jenkins Agent，构建完成后自动销毁，节省资源。
  2. 构建环境隔离与一致性： 每个构建都在独立的、预定义的 Pod 中运行，确保环境的纯净和一致性，避免“污染”和冲突。
  3. 资源利用率最大化： Kubernetes 负责调度 Agent Pod 到可用节点，优化集群资源利用。
  4. 云原生部署： 将应用程序打包成 Docker 镜像后，通过 Jenkins Pipeline 直接部署到 Kubernetes 集群。
  5. 自动化且可靠的部署： 利用 Kubernetes 的原生部署（Deployment）、服务发现（Service）、配置管理（ConfigMap/Secret）等能力，实现声明式、不可变的基础设施部署。
  6. 可观测性与回滚： 结合 Kubernetes 的滚动更新和回滚机制，Jenkins Pipeline 可以轻松实现零停机部署和快速回滚。

Jenkins 与 Kubernetes 集成方式

Jenkins 与 Kubernetes 的集成主要在两个层面：

  1. Jenkins Agent 运行在 Kubernetes Pod 中： Jenkins 将构建任务负载分发到动态创建的 Kubernetes Agent Pod 上。
  2. Jenkins Pipeline 部署应用到 Kubernetes： 构建好的应用（Docker 镜像）通过流水线部署到 Kubernetes 集群。

1. Jenkins Agent 运行在 Kubernetes Pod 中（弹性构建环境）

这是最推荐和最先进的 Jenkins Agent 管理方式。

a. 安装 Kubernetes 插件

  • 登录 Jenkins。
  • 前往 Manage Jenkins -> Manage Plugins。
  • 在 Available 标签页搜索并安装 “Kubernetes CLI” 和 “Kubernetes” 插件。

b. 配置 Kubernetes Cloud

  • 前往 Manage Jenkins -> Configure System。
  • 找到 Cloud 部分，点击 Add a new cloud -> Kubernetes。
  • Kubernetes URL： 填写您的 Kubernetes API Server URL。
    • 如果您 Jenkins Master 也在 Kubernetes 集群内运行，通常可以使用 https://kubernetes.default.svc.cluster.local。
    • 如果是外部集群，需要填写完整地址。
  • 凭据 (Credentials)： 选择或添加连接 Kubernetes 的凭据。最常见的是：
    • Kubernetes service account： 如果 Jenkins Master 运行在 K8s 中，可以使用本 Pod 的 Service Account。
    • Kubeconfig： 上传一个 Kubeconfig 文件。
    • Username with password / Secret text： 用于外部访问。
  • Jenkins URL： 填写 Jenkins Master 的外部可访问 URL，用于 Agent 连接 Master。
  • Jenkins tunnel： 如果需要 JNLP 协议的 Agent，配置隧道地址。
  • Pod Templates： 此处是核心配置，定义了 Jenkins Agent Pod 的不同规格。点击 Add Pod Template。
    • Name： 模板名称。
    • Labels： 关键！这是 Jenkins Job 选择 Agent 的标签，例如 maven-jdk11、nodejs-16。
    • Namespace： Agent Pod 将被创建到哪个 K8s Namespace。
    • Containers： 定义 Agent Pod 中的容器。
      • Container Name： 必须是 jnlp (如果使用 JNLP 协议连接) 或其他自定义名称。
      • Docker Image： 例如 maven:3.8.6-openjdk-11、node:16-alpine。
      • Command / Args： 通常留空，或根据镜像要求填写启动命令。
      • Resource Requests/Limits： 配置 CPU/内存请求和限制，确保资源隔离和调度。
      • Working Directory： /home/jenkins/agent 或 /var/jenkins_home/workspace。
      • Environment Variables / Volume Mounts： 配置其他环境变量或挂载所需的卷。
    • ServiceAccount： 为 Agent Pod 指定一个 ServiceAccount，该 ServiceAccount 需具备执行构建任务所需的 K8s 权限（例如，如果 Agent 需要执行 kubectl 命令来部署应用）。
  • 点击 **“保存” (Save)**。

最佳实践与注意事项

  • ServiceAccount 权限： Jenkins Agent Pod 所使用的 Kubernetes ServiceAccount 必须仅拥有执行其任务所必需的最小权限（最小权限原则）。
  • 凭据管理： 所有敏感信息（如 Docker Registry 凭据、Kubeconfig 内容）都应通过 Jenkins Credentials 进行管理，并通过 withCredentials 在 Jenkinsfile 中安全使用。
  • 版本管理： 确保 Kubernetes Manifests (YAML 文件) 和 Helm Charts 与应用程序代码一同进行版本控制。
  • 共享库： 强烈建议将复杂的 Kubernetes 部署逻辑封装在 Jenkins Shared Libraries 中。这可以使 Jenkinsfile 保持简洁，并促进团队内的CI/CD实践标准化和复用。
    • 例如，可以封装 deployAppToK8s(appName, imageTag, namespace) 等函数。
  • 回滚策略： Pipeline 应包含回滚机制，利用 Kubernetes 的内置能力（如 kubectl rollout undo）。
  • 环境隔离： 使用不同的 Kubernetes Namespace 或集群来隔离开发、测试、生产环境。
  • 可观测性： 部署后集成日志、监控和链路追踪系统，确保应用的健康和性能。
  • Jenkins Master 的部署： 考虑将 Jenkins Master 本身也部署在 Kubernetes 集群中（例如使用 Helm Chart），以充分利用 K8s 的高可用性、伸缩性和管理能力。

将 Jenkins 与 Kubernetes 深度集成，是实现高效、健壮和可扩展的云原生 CI/CD 流程的关键一步。它不仅简化了部署复杂性，还为您的团队带来了前所未有的敏捷性和控制力。
