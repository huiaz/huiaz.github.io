一、基本概念

理解 Docker，首先要掌握三个最核心的基本概念：镜像 (Image)、容器 (Container) 和 **仓库 (Registry)**。我们可以用一个简单的比喻来理解它们的关系：

  • 镜像 (Image)： 一个只读的模板，是创建容器的基础。

    • 比喻：可以把它想象成一个软件安装包，或者一个面向对象编程中的“类 (Class)”。
    • 技术细节：镜像内部包含了一个完整的文件系统，里面打包了应用程序运行所需的一切：代码、运行时库、环境变量、配置文件等。镜像是分层构建的，每一层都是对前一层的一组修改，这种分层结构使得镜像的复用和分发非常高效。

  • 容器 (Container)： 镜像的运行实例，是真正运行应用程序的地方。

    • 比喻：如果镜像是“类”，那么容器就是这个类的一个“实例 (Object)”。你可以用同一个镜像创建出任意多个相互隔离的容器。
    • 技术细节：容器本质上是宿主机上的一个特殊进程，但它拥有自己独立的命名空间（Namespace），包括进程、网络、文件系统等，实现了与宿主机和其他容器的隔离。容器在镜像的只读层之上增加了一个可写的“容器层”，所有在容器内发生的修改（如写入文件、修改配置）都发生在这个可写层，不会影响到底层的镜像。

  • 仓库 (Registry)： 集中存放和分发镜像的服务。

    • 比喻：可以把它看作是存放软件安装包的应用商店，或者存放代码的 GitHub。
    • 技术细节：仓库分为公共仓库和私有仓库。
      • **公共仓库 (Public Registry)**：最著名的是 Docker Hub，官方维护了大量的常用高质量镜像。
      • **私有仓库 (Private Registry)**：企业内部为了安全和速度，通常会搭建自己的私有仓库，如 Harbor、AWS ECR、Google GCR 等，用于管理公司内部的业务镜像。

三者关系总结： 我们从**仓库(Registry)拉取一个镜像(Image)到本地，然后以这个镜像为模板，运行一个或多个容器(Container)**来执行我们的应用。

二、核心组件

Docker 采用的是标准的 C/S (Client/Server) 架构。其核心组件包括 Docker 客户端 (Client)、Docker 守护进程 (Daemon) 和 REST API。此外，Dockerfile 也是构建镜像的核心工具。

  1. Docker 守护进程 (Docker Daemon / dockerd)

    • 角色：Docker 的“大脑”和“引擎”，是服务器端组件。
    • 功能：它是一个在宿主机上后台运行的进程，负责接收并处理来自客户端的请求。它承担了所有实际工作，包括：
      • 管理 Docker 对象：创建、运行、停止、删除容器。
      • 管理镜像：从仓库拉取镜像、在本地存储镜像、构建镜像。
      • 管理网络：创建和配置容器网络。
      • 管理数据卷：创建和管理容器的数据持久化。

  2. Docker 客户端 (Docker Client / docker)

    • 角色：用户的“遥控器”，是客户端组件。
    • 功能：这是用户与 Docker Daemon 交互的主要工具，通常以命令行界面 (CLI) 的形式存在。当你输入 docker run ... 或 docker build ... 这样的命令时，你就是在和 Docker 客户端交互。客户端会将这些命令转换成 REST API 请求发送给 Docker Daemon。

  3. REST API

    • 角色：客户端与守护进程之间的“通信语言”。
    • 功能：定义了客户端与守护进程之间通信的接口和协议。Docker 客户端通过这个 API 来指示守护进程执行各种操作。这也意味着，任何第三方工具只要遵循这套 API 规范，都可以与 Docker Daemon 进行交互，例如各种图形化管理工具（Portainer）或 CI/CD 系统。

  4. Dockerfile

    • 角色：构建镜像的“说明书”或“菜谱”。
    • 功能：它是一个文本文件，里面包含了一系列用户可以执行的指令。每一条指令都会在镜像中创建一个新的层。通过 docker build 命令，Docker Daemon 会读取 Dockerfile 的内容，并根据指令一步步地构建出一个自定义镜像。这是实现“基础设置即代码 (Infrastructure as Code)”的关键一环，保证了镜像构建过程的自动化、透明化和可复现。

工作流程示例

当你在终端输入 docker run nginx 命令时，整个流程是这样的：

  1. 客户端 (docker) 接收到你的命令。
  2. 客户端通过 REST API 将“运行一个基于 nginx 镜像的容器”的请求发送给本机的 **Docker 守护进程 (dockerd)**。
  3. 守护进程 收到请求后，首先检查本地是否存在 nginx 镜像。
  4. 如果本地没有，守护进程会连接到默认的 **仓库 (Docker Hub)**，拉取 nginx 镜像。
  5. 镜像下载完成后，守护进程会以这个镜像为模板，创建一个新的 容器。
  6. 容器启动后，守护进程会将容器的输出结果或状态返回给客户端。
  7. 客户端 将接收到的信息显示在你的终端上。
