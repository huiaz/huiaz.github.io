ReplicaSet (简称 RS) 和 ReplicationController (简称 RC) 的区别

从功能上讲，ReplicaSet 和 ReplicationController 的核心目标是一致的：

  • 确保指定数量的 Pod 副本始终在运行。
  • 当 Pod 发生故障、删除或节点失效时，它们会自动创建新的 Pod 以维持所需数量。
  • 当副本数量过多时，它们会终止多余的 Pod。

但它们之间存在关键的区别，主要体现在Selector (选择器) 的支持上。

1. ReplicationController (RC)

  • 出现时间： 这是 Kubernetes 最早引入的控制器之一 (Kubernetes v1.0 之前)。
  • 选择器 (Selector) 限制： 只支持基于等值（Equality-based）的选择器。这意味着你只能通过精确匹配标签的值来选择 Pod。
    • 例如：selector: app=nginx，它会选择所有带有 app: nginx 标签的 Pod。
    • 它不支持集合基（Set-based）的选择器，例如 app in (nginx, apache) 或 version notin (v1, v2)。
  • Pod 模板（Pod Template）： 负责创建新的 Pod，但不会创建与自身标签匹配的 Pod。
  • 用途： 随着 Kubernetes 的发展，RC 逐渐被弃用，被功能更强大的 ReplicaSet 取代。

2. ReplicaSet (RS)

  • 出现时间： Kubernetes v1.2 引入，作为 ReplicationController 的下一代和推荐替代品。
  • 选择器 (Selector) 功能增强： 支持集合基（Set-based）选择器。这是 RS 相对于 RC 最主要也是最重要的改进。
    • 例如：
      • selector: matchLabels: {app: nginx} （仍然支持等值匹配，因为等值匹配是集合匹配的子集）
      • selector: matchExpressions:
        • - {key: tier, operator: In, values: [frontend, backend]}
        • - {key: version, operator: NotIn, values: [v1beta1]}
        • - {key: environment, operator: Exists}
    • 这种更灵活的选择器能力使得 ReplicaSet 可以匹配更复杂的 Pod 标签条件，更好地管理 Pod。
  • Pod 模板（Pod Template）： 可以创建一个 Pod 模板，但不会创建与自身标签匹配的 Pod。
  • 独立使用： ReplicaSet 可以独立使用，但通常不建议直接独立使用。
  • 与 Desployment 的关系： ReplicaSet 主要由 Deployment 资源自动化管理。当我们创建一个 Deployment 时，Deployment 会在底层自动创建和管理 ReplicaSet。这是 ReplicaSet 最常见的用法。Deployment 负责滚动更新、回滚等高级功能，而 ReplicaSet 则负责维护Pod副本数量的稳定。

总结比较表格

特性 ReplicationController (RC) ReplicaSet (RS)
引入时间 早期版本 (v1.0 之前) v1.2 及之后
选择器支持 仅支持基于等值 (key=value) 支持等值和集合基 (key in (v1,v2), key exists)
功能 确保Pod副本数 确保Pod副本数
用途场景 已被取代，不推荐直接使用 通常由 Deployment 管理，极少直接独立使用
推荐程度 不推荐 推荐 (作为 Deployment 的一部分)

实际运维中的意义

作为运维工程师，你需要知道：

  1. 几乎所有情况下，你都应该使用 Deployment 而不是直接使用 ReplicaSet 或 ReplicationController。 Deployment 提供了自动化的滚动更新、回滚、暂停/继续等高级功能，这些功能是 ReplicaSet 和 ReplicationController 不具备的，但它们是现代化应用部署和管理所必需的。
  2. Deployment 在底层会自动为你创建和管理 ReplicaSet。 当你执行一次滚动更新时，Deployment 会生成一个新的 ReplicaSet 来管理新版本的 Pod，并逐步减少旧 ReplicaSet 的 Pod 数量，直到新版本完全取代旧版本。
  3. 了解 ReplicaSet 的存在对于排查问题很重要。 当 Deployment 出现问题时，你需要知道它背后是由 ReplicaSet 来控制 Pod 的数量和状态的。你可以通过 kubectl get rs 和 kubectl describe deploy <deployment-name> 来查看 Deployment 控制的 ReplicaSet 及其状态。

因此，理解 RC 和 RS 的区别，特别是 RS 强大的选择器以及它与 Deployment 之间的关系，对于高效地管理和排查 Kubernetes 集群中的应用程序是至关重要的。
