k8s Rolling Update

发表于2025-09-11|更新于2025-09-11|k8s面试题
|浏览量:

描述在 Kubernetes 中如何进行滚动更新和回滚操作。

🤔 分析过程：

此问题考察的是Kubernetes中最重要的功能之一：无停机更新和快速回滚。核心考察点是Deployment控制器如何通过管理ReplicaSet来实现这一过程。一个优秀的回答需要清晰地阐述”滚动更新”的内部机制（特别是maxSurge和maxUnavailable参数）、回滚是如何利用历史版本（旧的ReplicaSet）实现的，并能提供完整的命令行操作示例。

💡 答案生成：

1. 概念或定义

  • 滚动更新 (Rolling Update): 是Kubernetes中默认的部署策略，旨在实现零停机时间（Zero-Downtime）的应用升级。它通过逐个地用新版本的Pod替换旧版本的Pod，而不是一次性地销毁所有旧Pod再创建新Pod，从而保证在整个更新过程中，应用服务始终是可用的。

  • 回滚 (Rollback): 是一个恢复操作。当新版本的应用部署后出现问题时，回滚操作可以将应用快速地恢复到之前一个稳定、可用的版本。Kubernetes通过保留历史部署版本（以ReplicaSet的形式）来实现这一功能。

这两个功能都是由**Deployment**控制器来管理的。

2. 工作原理/机制

A. 滚动更新的工作原理

  1. 触发更新： 当你修改一个Deployment的Pod模板（Pod Template）时，例如更新了容器的镜像版本（spec.template.spec.containers[0].image），滚动更新就会被触发。
  2. 创建新ReplicaSet： Deployment控制器会创建一个新的ReplicaSet（代表新版本），并将其初始副本数设置为0。
  3. 逐个替换： 控制器会同时进行两个操作：
    • 增加新版本Pod： 缓慢地增加新ReplicaSet的副本数。
    • 减少旧版本Pod： 同时缓慢地减少旧ReplicaSet的副本数。
  4. 控制速率： 这个替换过程的速度和可用性由两个关键参数控制，它们定义在Deployment的spec.strategy.rollingUpdate字段下：
    • maxUnavailable：更新过程中，允许的不可用Pod的最大数量（或百分比）。例如，设置为1意味着在任何时候，至少要有 (期望副本数 - 1) 个Pod在提供服务。
    • maxSurge：更新过程中，允许超出的Pod的最大数量（或百分比）。例如，设置为1意味着在任何时候，Pod总数最多可以达到 (期望副本数 + 1) 个。
  5. 完成更新： 当所有旧版本的Pod都被终止，并且所有新版本的Pod都已成功创建并进入Ready状态时，更新过程完成。旧的ReplicaSet副本数变为0，但它不会被删除（除非达到历史版本限制），以便用于可能的回滚。

B. 回滚的工作原理

  1. 版本历史： Deployment会保留其管理过的ReplicaSet作为“修订历史”（Revisions）。每个ReplicaSet都代表了Deployment的一个历史版本。
  2. 触发回滚： 当你执行回滚命令时（例如 kubectl rollout undo）。
  3. 逆向操作： Deployment控制器会执行一个与滚动更新相反的过程。它会将当前的ReplicaSet（有问题的版本）的副本数逐渐减少，同时将上一个（或指定）版本的ReplicaSet的副本数逐渐增加，直到恢复到目标版本的状态。这个过程同样遵循maxSurge和maxUnavailable策略，确保回滚过程也是平滑的。

4. 扩展知识/最佳实践

  • 使用Readiness Probes： 这是确保零停机更新的关键！ Deployment在认为一个新Pod“可用”之前，会等待它的Readiness Probe成功。如果新版本的Pod有问题（例如启动失败或无法响应请求），它的Readiness Probe会失败，滚动更新就会自动暂停，从而防止有问题的版本完全接管流量，这给了你时间去调查和回滚。
  • 版本历史记录 (revisionHistoryLimit)： 在Deployment的spec中，可以设置revisionHistoryLimit字段（默认为10），用来指定保留旧ReplicaSet的数量。这有助于防止etcd中存储过多的历史记录。
  • 记录变更原因 (Change Cause)： 在使用kubectl apply或kubectl set image时，加上--record标志(已废弃, 但常考)可以记录命令到修订历史中，方便追溯。现在更推荐的做法是通过Git等版本控制系统来管理YAML文件，变更原因自然记录在commit message中，这是更规范的GitOps实践。

结论： Kubernetes通过Deployment控制器及其管理的ReplicaSet历史，提供了强大而灵活的滚动更新和回滚机制。配置好maxSurge、maxUnavailable和readinessProbe是实现安全、零停机应用生命周期管理的核心实践。
